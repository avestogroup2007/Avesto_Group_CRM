## Оценка текущего состояния

Система спроектирована грамотно. Это не «переписывать с нуля», а «усилить и почистить».

**Что уже хорошо (не трогаем):**

- Стек современный и правильный: Node.js + Express + Prisma + PostgreSQL + React/Vite
- Схема БД сильная: двойная запись (Дт/Кт), план счетов НСБУ РУз, идемпотентность через `refId`, неизменяемый журнал `TaskHistory`, аудит-лог
- Безопасность на месте: `helmet`, `cors` с credentials, rate-limit на логин, bcrypt, JWT, валидация окружения через `zod`
- Секреты только в окружении хостинга, не в коде
- Интеграция iiko изолирована в сервисный слой, ошибки настройки отдаются как 503

**Вывод по языкам:** менять язык НЕ нужно. Node.js спокойно тянет CRM на сотни одновременных пользователей. Проблема не в языке, а в структуре кода и нескольких узких местах. Мысль «перейти на более мощный язык» — распространённое заблуждение; переписывание убьёт полгода и принесёт новые баги, а не скорость.

## Приоритет 1 — Рефакторинг фронтенда (главное для развития)

### Проблема

`frontend/src/App.jsx` — **16 936 строк в одном файле**, более 60 компонентов, включая экраны по 1000–2000 строк:

| Компонент | Строк (примерно) | Назначение |
|---|---|---|
| CashRegisterView | ~2200 | Касса / инкассация |
| SalesAnalytics | ~1300 | Аналитика продаж |
| MoneyView | ~1180 | Казначейство |
| TopBar | ~800 | Верхняя панель |
| AutomationView | ~770 | Автоматизация |

**Почему это критично:**
- Файл медленно грузится у пользователя (весь код тянется сразу)
- Любая правка рискует сломать соседние экраны
- Невозможно работать в команде — постоянные конфликты
- Сложно найти нужное место

### Решение (пошагово)

1. **Настроить структуру папок:**
   ```
   frontend/src/
     components/     — переиспользуемые (Avatar, Badge, Kpi, Ring, Select...)
     pages/          — экраны (CashRegister, Money, SalesAnalytics, Tasks...)
     admin/          — админ-панель (AdminStaff, AdminBranches, AdminRoutes...)
     hooks/          — общая логика
     lib/            — api.js, утилиты, константы (цвета C, форматтеры)
     App.jsx         — только роутинг и раскладка (цель: < 300 строк)
   ```

2. **Внедрить `react-router-dom`** — сейчас навигация, судя по всему, через state (`view`/`setView`). Роутинг даст URL на каждый экран, кнопку «назад» браузера и ленивую загрузку.

3. **Ленивая загрузка (code splitting):** `React.lazy()` + `Suspense` для тяжёлых экранов. Пользователь грузит только тот экран, что открыл, а не все 17 тысяч строк сразу.

4. **Выносить по одному компоненту за раз**, проверяя работу после каждого. Начать с мелких переиспользуемых (Badge, Kpi, Field), потом крупные экраны.

**Порядок выноса (от простого к сложному):**
- Мелкие UI: Avatar, Badge, PhasePill, Kpi, Ring, Field, Select, NiceSelect, NiceDate
- Общие данные/константы: цвета, форматтеры, api-обёртки
- Экраны задач: Board, TaskCard, TaskDetail, CreateTask
- Финансовые экраны: MoneyView, CashRegisterView, PnlView, PostingsManager
- Аналитика: SalesAnalytics, Analytics, RiskyView
- Админка: весь блок Admin*

**Результат:** App.jsx ужимается с ~17000 до ~300 строк; каждый экран — отдельный файл 200–800 строк; первая загрузка ускоряется в разы.

## Приоритет 2 — Оптимизация БД и iiko (скорость)

### 2.1. N+1 запросы в синхронизации сотрудников

**Где:** `backend/src/services/iikoSync.js`, функция `syncEmployeesToDb`.

**Проблема:** в цикле по каждому сотруднику — отдельный `findUnique`, затем `update` или `create`. На 115 сотрудниках это ~230 запросов к базе за одну синхронизацию.

**Решение:**
- Один `findMany` со всеми `iikoId` заранее → построить Map в памяти
- Разделить на массивы «создать» и «обновить»
- `Promise.all` для параллельного выполнения, либо `createMany` для новых
- Итог: с ~230 запросов до ~3–5

### 2.2. iiko: авторизация на каждый запрос

**Где:** `backend/src/services/iikoServer.js`.

**Проблема:** каждый OLAP-запрос делает `auth` → запрос → `logout`. Три сетевых вызова там, где мог быть один. При построении отчёта с несколькими запросами — задержка растёт линейно.

**Решение:** кэшировать токен iiko в памяти с TTL (например, обновлять раз в N минут, а не на каждый вызов). Учесть ограничение iiko «одна сессия» — при параллельных запросах использовать общий токен или очередь.

### 2.3. Нет слоя кэширования тяжёлых отчётов

**Проблема:** отчёты продаж из iiko и агрегаты по деньгам пересчитываются при каждом открытии экрана.

**Решение:**
- Ввести кэш (для начала — простой in-memory с TTL; при росте — Redis)
- Кэшировать: OLAP-отчёты iiko за прошлые (закрытые) дни — они не меняются; агрегаты P&L и ДДС за закрытые месяцы
- Ключ кэша: период + филиал; инвалидация при новых проводках

### 2.4. Индексы БД — проверить покрытие

Схема уже содержит индексы на ключевых полях. Дополнительно проверить:
- Составные индексы под реальные запросы отчётов (`MoneyTx` по `date` + `direction` + `branchId` вместе)
- `Posting` по `date` + `debit`/`credit` для оборотно-сальдовой ведомости
- Прогнать `EXPLAIN ANALYZE` на самых частых запросах после накопления данных

### 2.5. Пагинация

Убедиться, что списки задач, движений денег и проводок отдаются постранично, а не целиком. При тысячах записей выдача всего списка положит и сервер, и браузер.

## Приоритет 3 — Инфраструктура и надёжность

Чтобы система стала «реальным рабочим инструментом без единой ошибки», нужен не только чистый код, но и защита от сбоев.

### 3.1. Тесты

Уже есть 4 тест-файла (auth, tasks, telegramBot) — отличная база. Расширить:
- Тесты на модуль денег (создание, согласование, баланс)
- Тесты на проводки (правила PostingRule → корректная пара Дт/Кт)
- Тесты на синхронизацию iiko (мок ответа сервера)
- Цель: покрыть критичную бизнес-логику, где ошибка = потеря денег или данных

### 3.2. CI/CD

- GitHub Actions: на каждый push — линт, тесты, сборка
- Автодеплой на Render только при зелёных тестах
- Это ловит ошибки до того, как они попадут к пользователям

### 3.3. Мониторинг и логи

- `pino` уже подключён — вывести логи в постоянное хранилище (не только консоль)
- Отслеживать ошибки (Sentry или аналог) — узнавать о сбоях раньше пользователей
- Health-check `/api/health` уже есть — подключить к аптайм-мониторингу

### 3.4. Резервное копирование БД

- Автоматический ежедневный бэкап PostgreSQL
- Проверенная процедура восстановления (бэкап без теста восстановления = нет бэкапа)
- Критично: в базе финансовые данные и кадры

### 3.5. Миграции

Prisma-миграции уже используются — хорошо. Правило: любое изменение схемы только через миграцию, никаких ручных правок боевой базы.

## Важные замечания

- **Не переписывать всё сразу.** Система работает — улучшать её нужно эволюционно, малыми проверяемыми шагами. «Большое переписывание» — главная причина, по которой рабочие системы умирают.
- **Каждый шаг — отдельный коммит** с проверкой, что ничего не сломалось.
- **Финансовая логика (двойная запись, план счетов НСБУ) требует проверки квалифицированным бухгалтером**, знакомым со стандартами Узбекистана, перед боевым запуском. Код может быть идеальным технически, но методология учёта — зона ответственности бухгалтера.
- **Резервные копии до любого крупного изменения.**
