# Архитектура Avesto CRM

Обзор устройства системы для разработчиков и будущей команды. Правила
разработки — в [STANDARDS.md](./STANDARDS.md).

---

## Общая картина

```
┌─────────────────────────────────────────────────────────────┐
│  BACK OFFICE (владелец продукта: owner, vendor)             │
│  Клиенты · продажи · MRR · Модули (вкл/выкл) · Самопроверка │
│  Развитие продукта · Мониторинг установок · Обратная связь  │
└─────────────────────────────────────────────────────────────┘
                    │ включает модули
                    ▼
┌─────────────────────────────────────────────────────────────┐
│  АДМИНКА (бизнес клиента: sysadmin, director)               │
│  Кадры · Филиалы · Доступ по ролям · Чек-листы (шаблоны)    │
│  SLA · Регламенты · Журнал безопасности · Резервная копия   │
└─────────────────────────────────────────────────────────────┘
                    │ настраивает
                    ▼
┌─────────────────────────────────────────────────────────────┐
│  ПРИЛОЖЕНИЕ (сотрудники клиента)                            │
│  Задачи · Деньги · Касса · Аналитика iiko · Чек-листы       │
│  Конструктор тортов · Производство · Telegram-бот (Mini App)│
└─────────────────────────────────────────────────────────────┘
```

Каждый клиент — **отдельная установка** (свой сервер, своя база). Back
Office живёт на установке владельца и НЕ имеет доступа к данным клиента:
только два односторонних канала — чтение `/api/health` (мониторинг) и
приём предложений (обратная связь). Настройки клиента меняются только в
его системе, его администратором.

## Стек

- **Бэкенд:** Node.js + Express + Prisma + PostgreSQL. JWT-аутентификация,
  helmet, CORS, rate-limit, zod-валидация, pino-логи, Sentry (опц.).
- **Фронтенд:** React + Vite + Tailwind. Разбит на `lib/` (домен),
  `components/` (UI), `pages/` (экраны), `admin/` (админка). Ленивая
  загрузка экранов, CSP в собранном html.
- **Интеграции:** iiko (продажи, ОПиУ, кадры, акты), Telegram (бот +
  уведомления), Claude API (ИИ-помощник).
- **Инфраструктура:** Render (Blueprint `render.yaml`), GitHub Actions CI
  (линт+тесты+сборка), ежедневный бэкап БД (Action) + бэкап из админки.

## Ключевые серверные модули

| Файл | Назначение |
|---|---|
| `services/orgConfig.js` | Конфигурация организации (бренд, юрлица, филиалы, окна) |
| `services/modules.js` | Реестр модулей продукта и флаги вкл/выкл |
| `routes/access.js` | Настройка доступа по ролям (оверрайды навигации) |
| `routes/selftest.js` | Самопроверка подсистем (Back Office) |
| `routes/vendor.js` | Back Office: клиенты, развитие, мониторинг, intake |
| `routes/backup.js` | Логическая резервная копия базы в JSON |
| `middleware/requireAuth.js` | Живая проверка учётки (кэш 60 сек) |
| `services/iikoServer.js` | Клиент iiko (кэш токена, single-flight) |
| `services/telegramBot.js` | Интерактивный бот (чек-листы, сводки, Mini App) |

## Конфигурации-одиночки (id=1, JSON)

- `OrgConfig` — организация (управляет sysadmin/director в админке)
- `ModuleConfig` — модули (управляет owner в Back Office)
- `AccessConfig` — доступ по ролям (управляет sysadmin в админке)
- `ApprovalConfig` — пороги согласования расходов (управляет director/sysadmin в админке)

Все читаются сервисом с кэшем TTL 60 сек и безопасными дефолтами.

---

## Модель доступа

1. `owner` проходит любые `requireRole` (владелец продукта).
2. `vendor` — только Back Office.
3. Бизнес-роли ограничены `requireRole` на маршрутах + видимостью разделов
   в меню (`navAllowed` в `lib/nav.js` с оверрайдами из `AccessConfig`).
4. Данные защищены на сервере независимо от меню: спрятанный пункт не
   даёт доступа к API.
