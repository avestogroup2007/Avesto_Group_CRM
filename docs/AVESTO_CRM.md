# Avesto Group CRM — документация системы

Система управления, финансового учёта и контроля персонала для сети
общественного питания. Единая панель для руководителя и офиса: заявки и задачи,
касса и деньги, бухгалтерия (двойная запись), зарплата, себестоимость, закупки и
склад, планы и KPI, чек-листы смен — с интеграцией **iiko** (источник истины по
продажам, кадрам, накладным и остаткам) и **Telegram-ботом** для персонала.

- **Валюта:** UZS (сум) · **Часовой пояс:** Asia/Tashkent
- **Юрлица сети:** «AVESTO CAFE» OK, «AVESTO SWEETS» OK, «INTERNATIONAL CATERING GROUP» MChJ
- **Развёртывание:** avesto.group/crm (собственный сервер) + превью на Render
- **Позиционирование:** универсальный, настраиваемый продукт («конструктор») —
  подходит любому бизнесу с товарным учётом, не только Avesto

**Ключевые архитектурные паттерны:**
- **Единый источник конфигурации в БД** (single-row JSON, id=1, кэш TTL 60 с):
  организация, доступ, пороги согласования, food-cost, ФОТ, отделы, закупки.
- **iiko как сменный адаптер:** все обращения к iiko инкапсулированы в сервисе;
  данные можно подменить другим источником.
- **Изоляция по филиалам** (`util/branchScope.js`): серверное ограничение видимых
  данных по назначенному филиалу сотрудника (не только видимость в UI).

## 3. Роли и доступ

| Роль | Назначение | Охват данных |
|------|-----------|--------------|
| **owner** | Владелец — полный доступ, обходит все ограничения | вся сеть |
| **director** | Директор — управление, финансы, настройки | вся сеть |
| **finance** | Финансы — деньги, отчёты, зарплата | вся сеть |
| **accountant** | Бухгалтер — финансы/отчёты (обзорная роль) | вся сеть (чтение) |
| **sysadmin** | Системный администратор — настройки, доступ | вся сеть |
| **manager** | Управляющий филиала | только свой филиал |
| **staff / cashier** | Линейный персонал, кассир | только свой филиал |
| **vendor** | Внешний партнёр (Back Office) | ограниченный |

**Механика доступа:**
- `requireAuth` каждую минуту перечитывает актуальную роль/филиал/активность из БД
  (блокировки и смена роли действуют без ожидания истечения токена).
- `requireRole(...)` — проверка роли на маршруте; **owner** всегда проходит.
- **Изоляция филиалов (fail-closed на чтении):** роль, привязанная к филиалу, но
  без назначенного филиала, **не видит** данные ни одного филиала (пока админ не
  назначит). Операции записи своей смены (касса/чек-лист) при этом не блокируются.
- Все чувствительные изменения (роли/доступ сотрудников, план счетов, пороги,
  справочники) пишутся в **журнал безопасности** (AuditLog).

## 4. Модули и функции

Меню сгруппировано по областям: **Задачи · Финансы · Аналитика · Персонал ·
Производство · Компания и настройки**.

### 4.1. Заявки и задачи
- **Входящие / Создать заявку** — жизненный цикл заявки из 5 фаз (Отправлено →
  Просмотрено → В работе → На проверке → Завершено).
- **Менеджер задач** — todo-доска: список и канбан, исполнители, сроки, важность,
  фильтры; счётчик активных/просроченных на дашборде; Telegram-напоминания.
- **Архив задач**, **Мои достижения** (KPI по закрытым задачам).

### 4.2. Финансы
- **Кассы** — сдача кассового отчёта по филиалу и дате (upsert), сверка с iiko,
  расчёт расхождения (недостача/излишек) с порогом алерта, подтверждение отчёта.
- **Учёт денег (MoneyTx)** — операции доходов/расходов, статьи (справочник),
  регулярные платежи, пороги согласования расходов (по общему порогу и по филиалу).
- **ДДС** — движение денежных средств по месяцам.
- **Бухгалтерия (двойная запись)** — план счетов (НСБУ Узбекистана), правила
  «статья → пара счетов», авто-формирование проводок из движений денег,
  оборотно-сальдовая ведомость (правка — директор/сисадмин + журнал).
- **ФОТ / Зарплата** — ставки в админке, помесячные часы/бонусы/штрафы,
  зарплатная ведомость.
- **Себестоимость (food cost)** — продажи по блюдам из iiko × заданная
  себестоимость (гибрид): ФК% и маржа по блюдам и итогам.
- **Согласование** — заявки на расход выше порога уходят на согласование.
- Экспорт реестра операций и ДДС в Excel/CSV.

### 4.3. Закупки и склад (модуль-«конструктор»)
Данные из iiko: приходные накладные, остатки, движение, взаиморасчёты.
- **Цены** — тренд закупочных цен с учётом **сезонной нормы**; сигналы «резкий
  рост / под наблюдением / сезонное падение».
- **Остатки** — текущий остаток, средний дневной расход, «хватит на N дней»,
  рекомендация к заказу; правила мин/макс (авто + ручные).
- **Движение** — сверка начало + приход − конец = расход за период.
- **Долги** — задолженность перед поставщиками:
  - тяга напрямую из iiko (**OLAP-отчёт по проводкам** «Задолженность перед
    поставщиками») с автоопределением полей;
  - импорт Excel-отчёта iiko «Задолженность перед контрагентами» (запасной путь);
  - **гибкий отчёт:** выбор периода (даты + пресеты), фильтр по торговому
    предприятию (филиалу), включение/исключение поставщиков галочками, разбивка
    по филиалам, просрочка.
- **Правила** — «конструктор» порогов сигналов, метод расчёта запаса, авто-проверка
  и отправка сигналов в Telegram (крон).

### 4.4. Аналитика
- **Аналитика продаж** — OLAP из iiko: по дням, способам оплаты, блюдам.
- **Планы и цели (план-факт)** — месячный план по филиалам, разбивка по дням
  (темп), % выполнения на дашборде.
- **Аналитика / Отчёты** — сводные показатели.
- **Дашборд руководителя** — сводка дня: касса по филиалам, деньги на согласовании,
  чек-листы, задачи, план-факт месяца, сигналы.

### 4.5. Персонал
- **Учёт времени (Timesheet)** — табель.
- **Чек-листы смены** — заполнение по шаблонам (в приложении и в Telegram),
  почасовые слоты, отчёт по филиалам и шаблонам, привязка к филиалу.
- **KPI сотрудников** — дисциплина по чек-листам.

### 4.6. Производство
- **Производство / Акт приготовления** — загрузка блюд/складов из iiko,
  предпросмотр и проведение акта приготовления в iiko (списание по тех.карте,
  оприходование готового блюда).
- **Конструктор** — конструктор изделий (напр. торты) + ИИ-помощник (Claude);
  позиционируется как универсальный настраиваемый модуль.

### 4.7. Компания и настройки
- **Оргструктура** — юрлица, филиалы, отделы; сопоставление филиалов с
  подразделениями iiko.
- **Back Office** — клиенты, продажи, развитие (для владельца).
- **Автоматизация** — крон-проверки, сигналы, напоминания.
- **Админ-панель** — доступ по ролям, справочники кассы, шаблоны чек-листов,
  пороги согласования, самопроверка системы.
- **О системе** — состав и версии.

## 6. Backend API

REST под `/api/*`, все маршруты за `requireAuth` (кроме крон-эндпоинтов с
секретом и вебхука Telegram). Группы и число операций:

| Группа | Назначение | Операций |
|--------|-----------|----------|
| `/api/auth` | вход, /me, смена пароля, выход | — |
| `/api/iiko` | прокси iiko: OLAP, ОПиУ, подозрительные, food-cost, сотрудники, акт | 12 |
| `/api/money` | учёт денег, статьи, регулярные, согласование | 18 |
| `/api/postings` | бухгалтерия: счета, правила, проводки, ОСВ | 13 |
| `/api/procurement` | закупки/склад: цены, остатки, движение, долги, правила | 13 |
| `/api/vendor` | Back Office (клиенты/продажи/развитие) | 10 |
| `/api/telegram` | настройки бота, темы, вебхук | 8 |
| `/api/tasks`, `/api/todos` | заявки и todo-доска | 5 + 5 |
| `/api/checklists`, `/api/checklist-templates` | чек-листы и шаблоны | 2 + 4 |
| `/api/cash`, `/api/staff`, `/api/payroll`, `/api/org` | касса, KPI, ФОТ, оргструктура | 3 каждая |
| `/api/dashboard`, `/api/plan`, `/api/foodCost`, `/api/approval`, `/api/access`, `/api/modules`, `/api/departments`, `/api/feedback` | сводка, план, food-cost, согласование, доступ, модули, отделы, обратная связь | 1–2 |
| `/api/audit`, `/api/backup`, `/api/selftest` | журнал, резервная копия, самопроверка | 1 |
| `/api/*-cron` | крон-эндпоинты (секрет в заголовке, constant-time) | 1 |

Ошибки обращения к iiko отдаются не как 500, а с понятной причиной (503 «не
настроено» / 502 «ошибка iiko»), чтобы интерфейс показывал текст.

## 7. Интеграция с iiko

Сервис `iikoServer.js` (18 экспортируемых функций): авторизация (login +
SHA1(pass) → ключ сессии, кэш 5 мин, single-flight), OLAP-отчёты продаж,
ОПиУ по плану счетов, подозрительные операции, food-cost, список сотрудников,
приходные накладные, остатки складов, взаиморасчёты, акт приготовления,
**долг по поставщикам через OLAP-проводки** (автоопределение полей отчёта под
конкретную сборку iiko + диагностика).

- Ключи и пароль iiko **не логируются**; все секреты — только в окружении сервера.
- Отчёты кэшируются (закрытые дни — долго, «сегодня» — коротко).
- Таймаут на тяжёлые запросы (синхронизация не «висит»).
- `iikoSync.js` — импорт сотрудников iiko в справочник CRM, правка доступа
  (роль/блокировка/филиал) с записью в журнал; повышение до owner/vendor запрещено.

## 8. Telegram-бот

Ролевое меню: руководству — сводки и касса по филиалам; персоналу — чек-листы и
быстрые действия с задачами (Мои задачи, закрыть, добавить). Mini App, темы
(topics) под разделы, напоминания о задачах и сигналы закупок. Вебхук защищён
секретом (сравнение постоянного времени).

## 9. Безопасность

- **JWT** (HS256, allowlist алгоритма) в httpOnly-cookie; свежесть сессии из БД.
- **bcrypt** для локальных паролей; iiko-SSO — по протоколу iiko.
- **helmet**, **CORS** (единственный origin, credentials), **rate-limit**
  (общий 300/мин, вход 20/15мин, ИИ 30/15мин).
- Секреты (DATABASE_URL, JWT_SECRET, IIKO_*, TELEGRAM_*, OWNER_LOGIN, крон-секреты)
  — **только в окружении**, не в коде/репозитории; `.env` в `.gitignore`.
- JWT **не пишутся в логи** (redact cookie/authorization).
- Импорт Excel — лимит размера + защита от битого файла (400, не 500);
  запись отчёта долга — в транзакции.
- **Изоляция филиалов** (fail-closed на чтении финансовых данных).
- **Журнал безопасности** на все чувствительные изменения.
- Нет SQL-инъекций (Prisma), XSS, SSRF (URL iiko только из окружения).

## 10. Развёртывание

- **Собственный сервер (прод):** `avesto.group/crm` — nginx отдаёт статику `/crm`
  и проксирует `/crm/api` → бэкенд (pm2), отдельная БД PostgreSQL. CRM **аддитивен**
  к существующему сайту (не пересекается). Обновление — одной командой:
  `bash deploy/update.sh` (git pull → миграции → сборка фронта → рестарт).
- **Render + превью** — авто-деплой из ветки `main` как «черновик».
- **Health-check:** `GET /api/health` (сервер + доступность БД).
