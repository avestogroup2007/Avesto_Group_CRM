# Стандарты разработки Avesto CRM

Этот документ — обязательные правила для всех, кто работает над системой
(владелец, разработчики, будущая команда). Цель: продукт развивается
предсказуемо, без ошибок, каждый шаг задокументирован и проверяем.

---

## 1. Архитектура: три уровня

Система строго разделена на три уровня. Смешивать их нельзя.

| Уровень | Роли | Назначение | Где живёт |
|---|---|---|---|
| **Back Office** | `owner`, `vendor` | Управление продуктом: клиенты, продажи, MRR, включение/выключение модулей, самопроверка, доска развития | Раздел «Back Office» |
| **Админка** | `sysadmin`, `director` | Настройка бизнеса клиента: кадры, филиалы, доступ по ролям, чек-листы, SLA | Раздел «Админ-панель» |
| **Приложение** | все бизнес-роли | Ежедневная работа: задачи, деньги, касса, аналитика, чек-листы | Основные разделы |

**Принцип:** новые функции сначала появляются как **модуль**, включаемый из
Back Office. Владелец тестирует, затем включает клиенту. Клиент настраивает
содержимое включённого модуля в админке. Сотрудник пользуется.

---

## 2. Роли

- `owner` — владелец системы. Полный доступ + Back Office. Проходит все
  проверки прав (`requireRole` пропускает owner всегда).
- `vendor` — команда продаж владельца. Только Back Office и «О системе».
- `director`, `finance`, `manager`, `accountant`, `sysadmin`, `staff` —
  бизнес-роли клиента.

Доступ к разделам настраивается сисадмином клиента (Админка → «Доступ по
ролям»); дефолты — в `frontend/src/lib/nav.js`.

---

## 3. Как добавить новую функцию (обязательный порядок)

1. **Модуль.** Если функция опциональна для клиентов — добавь ключ в
   `backend/src/services/modules.js` (`MODULES`), по умолчанию `false`.
2. **Модель БД.** Изменения схемы — ТОЛЬКО через миграцию Prisma
   (см. §5). Деньги — `BigInt`, не `Float`.
3. **API.** Новый роутер в `backend/src/routes/`, смонтировать в `app.js`.
   Обязательно: `requireAuth`, точные `requireRole`, `zod`-валидация тела,
   `asyncHandler`. Чувствительные действия — запись в `AuditLog`.
4. **Фронтенд.** Экран в `pages/` или `admin/`, лениво подключить в
   `App.jsx`, пункт меню в `lib/nav.js` (с гейтингом по модулю/роли).
5. **Тесты.** Интеграционный тест в `backend/test/` на права и логику
   (позитив + негатив: 403 для чужой роли, 400 на мусор).
6. **Самопроверка.** Если функция зависит от внешней настройки —
   добавь проверку в `backend/src/routes/selftest.js`.
7. **Документация.** Обнови `docs/PROJECT.md` (что за функция) и, если
   меняются правила, этот файл.

Каждый шаг — отдельный PR (см. §6). Ни один PR не мержится с красным CI.

## 4. Безопасность (не подлежит обсуждению)

- Секреты (iiko, Telegram, ANTHROPIC_API_KEY, VENDOR_*) — ТОЛЬКО в
  переменных окружения хостинга. Никогда в коде, коммитах, чатах.
- Каждый маршрут защищён `requireAuth`; данные бизнеса — `requireRole`.
- Живая проверка учётки (`requireAuth`): заблокированный сотрудник
  теряет доступ в течение минуты.
- Валидация всех входных данных через `zod`, с ограничением длины/диапазона.
- Двойная запись денег (Дт/Кт) — только в транзакции; правки согласованных
  сумм возвращают операцию на согласование и пишутся в аудит.
- Внешние URL (мониторинг) — с защитой от SSRF (блок внутренних адресов,
  `redirect: "manual"`).
- Секреты-токены сравниваются за постоянное время (`timingSafeEqual`).

## 5. База данных

- Любое изменение схемы — миграцией:
  `npx prisma migrate diff --from-schema-datasource prisma/schema.prisma \
   --to-schema-datamodel prisma/schema.prisma --script` → в файл
  `prisma/migrations/<timestamp>_<name>/migration.sql`, затем
  `npx prisma migrate deploy && npx prisma generate`.
- Ручные правки боевой базы запрещены.
- Конфигурации-одиночки (организация, модули, доступ) — модель с `id=1`
  и полем `data Json`, кэш в сервисе с TTL 60 сек.

---

## 6. Процесс и качество

- Работа — на ветке `claude/...`; каждая задача — отдельный PR.
- Перед коммитом: `npm run lint`, `npm run format:check` (бэкенд),
  `npm run build` (фронтенд), `npm test` (все тесты зелёные).
- CI (GitHub Actions) гоняет линт+формат+тесты (бэкенд) и сборку (фронт)
  на каждый PR. Мерж — только по зелёному CI, метод squash.
- Комментарии в коде — по-русски, объясняют «почему», а не «что».
- Резервная копия базы — перед любым крупным изменением
  (Админка → Система → «Скачать резервную копию»).

---

## 7. Проверка на ошибки (регулярно)

- Полный аудит перед сдачей клиенту: статический разбор нового кода +
  живой прогон API + Playwright-обход экранов на desktop и mobile.
- Самопроверка (Back Office → «Самопроверка») — после каждой настройки
  установки.
- Финансовая логика (двойная запись, план счетов НСБУ РУз) требует
  проверки квалифицированным бухгалтером перед боевым запуском.
