# Avesto Group CRM System — документация проекта

CRM/BPM-система для сети заведений Avesto Group (Узбекистан):
«AVESTO CAFE» OK, «AVESTO SWEETS» OK, «INTERNATIONAL CATERING GROUP» MChJ.
Валюта — сум (UZS), часовой пояс — Asia/Tashkent. Источник правды по кадрам
и товарам — iiko.

- **Фронтенд (прод):** https://avestogroup2007.github.io/Avesto_Group_CRM/
- **Бэкенд (прод):** https://avesto-crm-backend.onrender.com
- **Репозиторий:** https://github.com/avestogroup2007/Avesto_Group_CRM

## 3. Модули системы (разделы приложения)

| Раздел | Что делает |
|---|---|
| Сводка дня (дашборд) | Экран руководителя «как идут дела сегодня»: касса по филиалам и расхождения с iiko (недостача/излишек), расходы на согласовании, % чек-листов, алерты, а также блок «План месяца» (% выполнения плана выручки и темп опережения/отставания к сегодня из `/api/plan`) и счётчик задач команды (активные/просроченные из менеджера задач, с переходом в раздел; просрочка — в алерты). Собирает уже имеющиеся данные (`/api/dashboard`), управляющий видит свой филиал |
| KPI сотрудников | Рейтинг линейного персонала по чек-листам за период (сдачи, средний % выполнения, активные дни, последняя активность). Реальные серверные данные (`/api/staff/kpi`), управляющий видит свой филиал |
| Менеджер задач | Удобная доска дел (в стиле Google Tasks), отдельно от формального процесса заявок: быстрый ввод, чистый список с чекбоксами/сроками/исполнителями/звёздочкой «важное», канбан-доска (перетаскивание между статусами К выполнению/В работе/Готово), фильтры (исполнитель, филиал, важные) и поиск. Данные серверные (`/api/todos`) — задачи общие для команды, охват по ролям/филиалу; правят автор, исполнитель или офис. Telegram-напоминания о просроченных/сегодняшних задачах исполнителю раз в день (крон-эндпоинт `POST /api/todos-cron/remind` с секретом `TODO_REMINDER_SECRET`, дедуп за день). Быстрые действия в боте: «📋 Мои задачи» — список назначенных, закрытие кнопкой и добавление новой одним сообщением |
| Входящие / Создать заявку | Задачи и заявки: ручная форма (исполнитель, контролёр, сумма, срок), маршруты согласования, SLA-таймеры, фазы (новая → в работе → на проверке → закрыта), SOP-чек-листы с фото-гейтом |
| Мои достижения | Личный кабинет сотрудника: данные из iiko, отработанные часы/дни, эффективность, смена пароля |
| Аналитика | Кабина директора: воронка задач, просрочки, нагрузка |
| Учёт времени | Смены, табель |
| Кассы | Кассовые отчёты по 4 филиалам с кассами (Микрорайон, Узбекистанская, Аэропорт, Наврузий Магазин): сверка с iiko, сейф филиала, инкассация в офис с подтверждением |
| Учёт денег | Движения денег, заявки на расход с согласованием (директор/финансы). Порог согласования настраивается в админке (общий и по филиалам): расход не выше порога проводится сразу, выше — на согласование; порог 0 — согласуется любой расход. Регулярные операции (авто-создание по расписанию, идемпотентно), бюджеты филиалов |
| ДДС (движение денег) | Отчёт «движение денежных средств» по месяцам и статьям ДДС: приток/отток и чистый поток за период (только согласованные операции). Данные из движений денег (`/api/money/dds`). Выгрузка в CSV (Excel): ДДС и реестр операций (`/api/money/export`) |
| ФОТ / Зарплата | Зарплатная ведомость по месяцам: ставка на сотрудника (оклад или почасовой) задаётся в системе (в iiko-API ставок нет), помесячные часы/бонусы/штрафы, итог к выплате считается автоматически, выгрузка в CSV. Данные серверные (`/api/payroll`); ставки и записи меняют директор/сисадмин, читают финансы/бухгалтер. Авто-подстановка часов из iiko-посещаемости — следующий шаг |
| Себестоимость (food cost) | Продажи по блюдам из iiko × себестоимость: по каждому блюду выручка, себестоимость, food cost % и маржа, сверху итоги, выгрузка CSV. Гибрид цены: заданная в системе цена за единицу блюда → % по группе блюд → общий % по умолчанию (авто-подстановка себестоимости из тех.карт iiko — следующий шаг). Отчёт `/api/iiko/food-cost` (продажи из iiko), настройки `/api/food-cost/config` (правят директор/сисадмин, читают финансы/бухгалтер) |
| Планы и цели (план-факт) | Месячный план выручки и расходов по филиалам против факта: факт выручки — сверённая с iiko выручка касс за месяц, факт расходов — согласованные расходы «Учёта денег». Разбивка по дням даёт темп (опережение/отставание к сегодня), % выполнения плана, итоги по сети. Данные серверные (`/api/plan`); план правят директор/сисадмин, читают управленческие роли (управляющий — свой филиал) |
| Проводки | Двойная запись Дт/Кт: план счетов, правила автопроводок из движений денег, оборотно-сальдовая ведомость (Дт = Кт) |
| Аналитика продаж | Живые данные iiko: выручка по дням/филиалам, типы оплат, топ блюд, часы, официанты, PnL |
| Производство | Акт приготовления iiko: справочники (блюда с тех.картой, склады), предпросмотр XML, проведение (списание ингредиентов + оприходование) |
| Чек-листы смены | Почасовой санитарный обход (окно: производство 07–16, рестораны 08–20), пункты с обязательным фото, открытие/закрытие смены, статусы часов. Плюс чек-листы по шаблонам из админки: по должности (role) и уборка по часам (cleaning) — сотрудник заполняет свои, сдачи пишутся с ссылкой на шаблон; руководству — отчёт за период (`/api/checklists/report`). Модули включает владелец в Back Office |
| Торты (конструктор) | Пошаговый мастер: основа (фиксируется 🔒) → покрытие → украшения (без ограничений). Себестоимость по граммовке (вес × цена за кг). Стандарты привязаны к позициям iiko. ИИ-помощник подбирает состав по описанию заказа |
| Отчёты | Сводные отчёты |
| Автоматизация | Автоправила по задачам; Telegram: тест, темы, помощник подключения, бот чек-листов (вебхук) |
| Оргструктура | Компании, филиалы, должности. «Мастер настройки» (первый запуск): пошагово бренд → юрлица → филиалы (подразделение iiko, касса/производство, окно смены) → сохранение в конфигурацию организации (`/api/org`). Открывается один раз директору/сисадмину и по кнопке на экране; веб и Telegram-бот сразу видят изменения |
| Админ-панель | Сотрудники (учётки из iiko: роль, рабочий филиал, вход, привязка Telegram), должности, филиалы/бюджеты, отделы, маршруты, SLA, регламенты, доступ по ролям, чек-листы, справочники кассы (статьи прихода/расхода, ДДС, типы оплат, юрлица — общие с разделом «Деньги», изменения пишутся в журнал безопасности), система |

**Рабочий филиал сотрудника.** В колонке «Филиал и бот» админ закрепляет
сотрудника за филиалом (поле `checklistBranch`). Привязанный сотрудник видит
в приложении только данные своего филиала (задачи, касса, чек-листы) и
проходит чек-листы этого филиала в боте. Старшие роли (руководство, финансы,
сисадмин) выбирают филиал сами; без привязки — доступ по роли, без
ограничения. Филиал отдаётся в `/api/auth/login` и `/api/auth/me` (поле
`branch`).

Ограничение по филиалу — не только видимость в UI, но и граница на СЕРВЕРЕ
(`util/branchScope.forcedBranch`): привязанный сотрудник не может ни записать,
ни прочитать данные чужого филиала (`/api/checklists/run`, `/api/cash/report`,
`/api/cash/reports`, `/api/checklists/report`, iiko OLAP продажи/ОПиУ/риски).
**Исключение — бухгалтер:** по решению владельца он видит финансы по всей
компании (обзорная роль), тогда как управляющий ограничен своим филиалом.

## 4. Интеграции

### 4.1 iiko (iikoServer / resto API)
- Авторизация: `/resto/api/auth?login=..&pass=SHA1(пароль)` → ключ → `logout`.
- Продажи: OLAP-отчёты (выручка, оплаты, блюда, часы, официанты), PnL.
- Кадры: сотрудники → синхронизация в User (уволенные блокируются; линейный
  персонал создаётся неактивным — доступ открывает администратор).
- Производство: справочники (`products/list`, `corporation/stores`), акт
  приготовления через `documents/import/productionDocument` (XML), dry-run.
- SSO: пароль сотрудника проверяется живьём в iiko при входе (с кэшем bcrypt
  на случай недоступности iiko).

### 4.2 Telegram
- Один бот, одна операционная супергруппа, уведомления раскладываются по
  темам (`message_thread_id`): Расходы, Заявки, Касса, Персонал, Отчёты,
  Чек-лист.
- Авто-алерт расхождения кассы: при сдаче кассового отчёта сервер сверяет
  сумму по видам оплат с выручкой iiko и при расхождении выше порога
  (1000 сум) шлёт в тему «Касса» уведомление о недостаче/излишке.
- Интерактивный бот чек-листов (вебхук `/api/telegram/webhook`, секрет в
  заголовке): сотрудник входит по логину/паролю из CRM (SSO iiko), выбирает
  час обхода, отмечает пункты кнопками, присылает фото (обязательно для
  санитарных пунктов), итог сохраняется в БД и сводкой с фото уходит в тему.
  Помимо легаси-обходов бот показывает чек-листы по шаблонам из админки: по
  должности (role, отфильтрованы по должности сотрудника) и уборку по часам
  (cleaning, разворачивается в подменю часов). Кнопки видны только при
  включённом владельцем модуле; сдачи пишутся с ссылкой на шаблон и попадают
  в тот же отчёт, что и веб.
- Помощник подключения: getMe/getUpdates, поиск chat_id и id тем, «Проверить
  темы» — тестовое сообщение в каждую настроенную тему.

### 4.3 Claude API (ИИ)
- Официальный SDK `@anthropic-ai/sdk`, модель `claude-opus-4-8`.
- `POST /api/ai/cake-suggest`: по описанию заказа подбирает состав торта
  строго из стандартов (строгий JSON через `output_config.format`).
- Ключ `ANTHROPIC_API_KEY` — только в окружении сервера.
- План: помощники в акте приготовления и аналитике.

## 6. База данных (Prisma, PostgreSQL)

Основные модели: `Company`, `Branch`, `User` (кадры из iiko: iikoId, login,
telegramId, checklistBranch, роли director/finance/manager/accountant/
sysadmin/staff), `Task`/`TaskHistory`/`Comment`, `CashReport`, `MoneyTx`
(идемпотентность по refId), `MoneyDict`, `MoneyRecurring`, `LedgerAccount`/
`PostingRule`/`Posting` (двойная запись), `AuditLog`, `ShiftChecklistRun`
(чек-листы из бота), `BotSession` (диалог бота).

Деньги хранятся в BigInt (сумы), сериализация через строку.

### Размещение на своём сервере под подпутём (`avesto.group/crm`)

Полная пошаговая инструкция — в `deploy/DEPLOY.md` (+ готовые конфиги
`deploy/nginx-avesto-crm.conf`, `deploy/Caddyfile`, юнит
`deploy/avesto-crm.service`, пример окружения `backend/.env.production.example`).
Кратко: фронт и API на одном домене (same-origin), reverse-proxy отдаёт статику
под `/crm` и проксирует `/crm/api → :3000`.

- Сборка фронта: `VITE_BASE=/crm/ VITE_API_URL=/crm npm run build`.
- Окружение бэкенда: `FRONTEND_URL=https://avesto.group`, `COOKIE_SECURE=true`,
  `COOKIE_SAMESITE=lax` (same-origin), `PUBLIC_BASE_URL=https://avesto.group/crm`
  (Telegram-вебхук), `PUBLIC_APP_URL=https://avesto.group/crm/` (кнопка «Открыть
  CRM»).

## 8. Аутентификация и доступы

- Вход только для сотрудников из iiko (source=iiko, не уволен) + bootstrap-
  админ; grace-режим до первой синхронизации.
- Пароль проверяется живьём в iiko (SSO) с локальным bcrypt-кэшем.
- Первый вход — обязательная смена временного пароля (табельный номер).
- Линейный персонал (официанты, охрана и т.п.) создаётся без доступа к CRM;
  доступ конкретному человеку открывает администратор.
- Telegram-бот: самостоятельная привязка по логину/паролю (сообщение с
  паролем бот удаляет из чата), один Telegram — один сотрудник.

## 9. Процессы разработки

- Ветка разработки: `claude/new-session-2bbxx0` → PR → CI (бэкенд: линт,
  формат, тесты; фронтенд: сборка) → squash-merge в `main` → автодеплой
  Render + GitHub Pages.
- Тесты: node:test, интеграционные (auth, tasks, telegram-бот) — общая
  тестовая БД, тесты создают своих пользователей.
- Стиль: Prettier + ESLint, комментарии на русском.
